## 1. 문법은 목적이 아니라 현실을 번역하기 위한 도구이다

프로그래밍을 처음 배우는 사람은 대개 변수, 자료형, 조건문, 반복문, 함수, 리스트, 딕셔너리, 클래스, 예외 처리, 파일 입출력, 라이브러리의 순서로 문법을 배운다. 이 순서는 틀리지 않다. 그러나 이 순서만 따라가면 "문법은 아는 것 같은데, 막상 무언가를 만들려고 하면 손이 멈추는 상태"에 빠지기 쉽다.

그 이유는 문법을 암기 대상으로 배웠을 뿐, 현실 문제를 표현하는 도구 체계로 배우지 않았기 때문이다. 문법 학습의 진짜 목적은 현실의 문제를 컴퓨터가 처리할 수 있는 형태로 번역하는 능력을 얻는 데 있다.

프로그래밍은 `if`, `for`, `class`를 아는 일이 아니라, 현실의 사람, 거래, 주문, 계좌, 물류 흐름, 법률 판단, 위험 신호를 데이터, 조건, 반복, 함수, 객체, 테이블, 규칙, 화면으로 바꾸는 작업이다.

따라서 프로그래밍 학습은 "문법을 얼마나 많이 알고 있는가"가 아니라 "현실의 문제를 어떤 자료구조, 알고리즘, 함수, 객체, 인터페이스로 바꿀 수 있는가"로 평가되어야 한다.

## 2. 자연어와 프로그래밍 언어의 결정적 차이

자연어는 주로 세계를 서술한다. 반면 프로그래밍 언어는 세계를 서술할 뿐 아니라 실행하고, 그 실행을 통해 컴퓨터 내부의 상태를 변화시킨다.

예를 들어 "파산자가 파산 직전에 가족에게 여러 차례 소액을 이체했다"라는 자연어 문장은 현실을 설명한다. 독자는 이 문장을 보고 재산 은닉 가능성, 편파 변제 가능성, 수취인 관계 확인 필요성을 추론할 수 있다. 그러나 컴퓨터는 이 문장을 그대로 처리하지 못한다.

```python
rule = SplitTransferRule(
    window_days=30,
    min_count=5,
    total_amount_threshold=5_000_000
)
result = rule.evaluate(debtor_transactions)
```

위 코드는 단순 설명이 아니다. 이 코드는 거래 데이터를 검사하고, 조건에 맞는 거래를 찾고, 합계를 계산하고, 반복 송금 여부를 확인하고, 결과를 반환한다. 따라서 프로그래밍을 배울 때의 핵심 질문은 "이 문법이 무슨 뜻인가"에서 멈추면 안 된다.

**핵심 질문은 "이 코드가 실행되면 컴퓨터 안의 상태가 어떻게 바뀌는가"이다.**

초보자는 코드를 문장처럼 읽는다. 중급자는 코드를 상태 변화의 연쇄로 읽는다. 숙련자는 코드가 만들어내는 데이터 흐름, 책임 분리, 추상화 구조, 시스템 경계를 함께 본다.

## 3. 프로그래밍의 기본 뼈대: 순차, 조건, 반복

거의 모든 프로그램은 순차, 조건, 반복이라는 세 가지 제어 흐름 위에 서 있다. 여기에 자료구조가 살을 붙이고, 함수와 클래스가 구조를 만든다.

| 핵심 흐름 | 의미 | 예시 |
|---|---|---|
| 순차 | 정해진 순서대로 실행한다 | 데이터를 불러오고, 정제하고, 분석한다 |
| 조건 | 상황에 따라 다른 길로 간다 | 금액이 기준 이상이면 의심 거래로 표시한다 |
| 반복 | 여러 대상을 차례로 처리한다 | 모든 거래 내역을 하나씩 검사한다 |

```python
for transaction in transactions:
    if transaction.amount >= threshold:
        mark_as_suspicious(transaction)
```

이 코드는 단순하지만 많은 FDS 시스템의 기본 골격이다. 다만 실제 프로그램은 단순히 반복하고 조건을 검사하는 차원을 넘어 상태, 자료구조, 추상화, 인터페이스를 반드시 다루어야 한다.

| 추가 핵심 | 의미 |
|---|---|
| 상태 | 시간이 지나며 값이 어떻게 변하는가 |
| 자료구조 | 데이터를 어떤 모양으로 담을 것인가 |
| 추상화 | 복잡한 것을 어떤 이름과 단위로 묶을 것인가 |
| 인터페이스 | 다른 코드, 데이터베이스, 사용자와 어떻게 연결될 것인가 |

## 4. 상태: 프로그램은 실행되면서 세계를 바꾼다

상태란 현재 값이 무엇인지를 뜻한다. 프로그램은 실행되면서 상태를 읽고, 조건에 따라 상태를 바꾸며, 바뀐 상태를 다음 판단의 근거로 사용한다.

```python
risk_score = 0

# 반복 송금 정황 발견
risk_score += 30

# 가족 또는 특수관계인 송금 정황 발견
risk_score += 20

# 파산 신청 직전 30일 이내 거래 확인
risk_score += 25
```

위 예시에서 `risk_score`는 처음부터 정해진 값이 아니다. 프로그램이 데이터를 읽고, 조건을 검사하고, 판단을 누적하면서 상태가 변화한 결과이다.

| 상태의 예 | 의미 |
|---|---|
| 현재 위험 점수 | 지금까지 발견된 의심 정황의 강도 |
| 누적 송금액 | 특정 기간 동안 특정 수취인에게 보낸 총액 |
| 거래 횟수 | 반복성 판단의 근거 |
| 마지막 거래 시점 | 파산 신청일과의 거리 |
| 조사 완료 여부 | 분석 프로세스의 진행 상태 |

코드를 볼 때는 항상 "이 코드는 어떤 값을 읽는가", "어떤 조건을 검사하는가", "어떤 값을 바꾸는가", "바뀐 값은 다음 판단에 어떻게 영향을 주는가"를 질문해야 한다.

## 5. 자료구조: 현실을 어떤 렌즈로 볼 것인가

입문 단계에서는 리스트, 딕셔너리, 튜플, 집합을 단순한 데이터 그릇으로 배운다. 그러나 중급 이후에는 그렇게 보면 부족하다. 자료구조는 데이터를 담는 그릇이 아니라, 현실을 어떤 관점에서 볼 것인가를 결정하는 렌즈이다.

### 5.1 리스트로 보면 순서가 보인다

```python
amounts = [100000, 200000, 150000, 300000]
```

리스트는 순서 있는 사건 목록을 표현하기에 적합하다. 거래 금액의 흐름, 특정 시점 이후 패턴 변화, 반복적으로 등장하는 금액 등을 확인할 때 유용하다.

### 5.2 딕셔너리로 보면 관계가 보인다

```python
transactions_by_receiver = {
    "김OO": [100000, 200000, 150000],
    "이OO": [300000, 500000],
}
```

딕셔너리는 key-value 구조를 통해 관계와 매핑을 표현한다. 수취인별 거래, 고객별 위험 점수, 상품별 매출, 파산자별 의심 거래 횟수 등을 다룰 때 적합하다.

### 5.3 테이블로 보면 분석이 가능해진다

```python
transactions = [
    {"debtor_id": "D001", "receiver": "김OO", "amount": 100000, "date": "2026-01-01"},
    {"debtor_id": "D001", "receiver": "김OO", "amount": 200000, "date": "2026-01-03"},
    {"debtor_id": "D001", "receiver": "이OO", "amount": 500000, "date": "2026-01-10"},
]
```

이 구조는 이후 pandas DataFrame으로 바꾸기 좋다. 그룹화, 집계, 필터링, 시각화를 수행하기에 적합하다.

```python
df.groupby("receiver")["amount"].sum()
```

이 한 줄은 "수취인별 총 송금액을 계산하라"는 현실의 조사 행위를 코드로 번역한 것이다.

### 5.4 그래프로 보면 자금 흐름 네트워크가 보인다

FDS가 고도화되면 거래를 단순한 행 목록이 아니라 네트워크로 본다. 파산자, 수취인, 계좌, 법인을 노드로 두고, 송금 관계를 엣지로 표현하면 자금의 이동 경로를 분석할 수 있다.

| 관점 | 보이는 것 | 적합한 질문 |
|---|---|---|
| 리스트 | 순서와 흐름 | 거래가 어떤 순서로 발생했는가 |
| 딕셔너리 | 관계와 매핑 | 누구에게 얼마가 집중되었는가 |
| 테이블 | 집계와 분석 | 조건별 금액과 횟수는 어떻게 다른가 |
| 그래프 | 네트워크와 경로 | 돈이 어떤 경로를 따라 이동했는가 |

## 6. 추상화: 복잡한 현실을 이름 붙여 다룰 수 있게 만드는 기술

현실은 복잡하다. 파산관재인이 보는 현실에는 채무자, 채권자, 가족, 지인, 계좌, 송금, 현금 인출, 부동산 처분, 자동차 이전, 신용카드 사용, 파산 신청일, 면책 신청, 편파 변제, 사해행위, 재산 은닉 등이 함께 존재한다.

이 모든 것을 코드에 그대로 흩뿌리면 프로그램은 금방 망가진다. 추상화란 복잡한 현실을 의미 있는 단위로 묶고 이름을 붙이는 일이다.

```python
class Debtor:
    pass

class Transaction:
    pass

class Receiver:
    pass

class FraudRule:
    pass

class Explanation:
    pass
```

| 현실 개념 | 코드 추상화 |
|---|---|
| 파산자 | `Debtor` |
| 거래 | `Transaction` |
| 수취인 | `Receiver` |
| 의심 규칙 | `FraudRule` |
| 쪼개기 송금 | `SplitTransferRule` |
| 설명 가능한 판단 | `Explanation` |
| 관재인 검토 보고서 | `InvestigationReport` |

좋은 프로그래머는 코드를 많이 치는 사람이 아니다. 좋은 프로그래머는 현실을 잘 나누고, 좋은 이름을 붙이고, 각 단위의 책임을 분명히 하는 사람이다.

## 7. 인터페이스: 프로그램은 혼자 존재하지 않는다

입문자는 코드를 하나의 파일 안에서만 생각한다. 그러나 실제 프로그램은 항상 외부와 연결된다. 데이터는 외부에서 들어오고, 결과는 사용자나 다른 시스템으로 나간다.

| 연결 대상 | 인터페이스 |
|---|---|
| 거래 데이터 | CSV, Excel, 데이터베이스 |
| 사용자 | Streamlit 대시보드 |
| 분석 로직 | Python 함수, 클래스 |
| 저장소 | PostgreSQL, SQLite |
| 보고서 | PDF, Excel, HTML |
| 외부 시스템 | API |
| 관재인 화면 | 표, 그래프, 설명 문장 |

프로그램 설계는 결국 "데이터는 어디서 들어오는가", "어떤 구조로 저장되는가", "어떤 함수와 규칙이 판단하는가", "결과는 어디로 나가는가", "사용자는 무엇을 보고 어떤 결정을 내리는가"를 정의하는 일이다.

```
거래 데이터 입력
    ↓
데이터 정제
    ↓
자료구조 변환
    ↓
의심 규칙 적용
    ↓
위험 점수 계산
    ↓
설명 가능한 근거 생성
    ↓
Streamlit 대시보드 표시
    ↓
관재인 검토 및 법률 판단
```

## 8. 현실 문제와 코드 사이의 번역 사고틀

프로그래밍을 잘하려면 현실을 바로 코드로 쓰려고 하면 안 된다. 현실과 코드 사이에는 번역 단계가 필요하다. 숙련된 프로그래머는 대개 현실 문장, 명사, 동사, 조건, 반복 대상, 결과, 사용자 화면의 순서로 문제를 분해한다.

### 8.1 현실 문장을 쓴다

먼저 사람이 이해하는 말로 문제를 쓴다. 예를 들어 "파산자가 파산 신청 전 30일 동안 동일 수취인에게 5회 이상 송금했고, 총액이 500만 원을 넘으면 쪼개기 송금으로 의심한다"라고 쓴다.

### 8.2 명사를 찾는다

| 명사 후보 | 코드 요소 |
|---|---|
| 파산자 | `Debtor` |
| 파산 신청일 | `bankruptcy_filing_date` |
| 수취인 | `Receiver` |
| 송금 | `Transaction` |
| 30일 | `window_days` |
| 5회 | `min_count` |
| 500만 원 | `total_amount_threshold` |
| 쪼개기 송금 | `SplitTransferRule` |

### 8.3 동사를 찾는다

| 동사 후보 | 코드 요소 |
|---|---|
| 송금했다 | `create transaction / transfer` |
| 계산한다 | `calculate_total_amount` |
| 센다 | `count_transactions` |
| 비교한다 | `compare_with_threshold` |
| 의심한다 | `mark_as_suspicious` |
| 설명한다 | `generate_explanation` |

### 8.4 조건과 반복 대상을 찾는다

| 현실 조건 | 코드 조건 |
|---|---|
| 30일 동안 | `date >= filing_date - 30days` |
| 동일 수취인 | `groupby("receiver")` |
| 5회 이상 | `count >= 5` |
| 총액 500만 원 초과 | `total_amount >= 5_000_000` |

| 반복 대상 | 코드 표현 |
|---|---|
| 모든 거래 | `for transaction in transactions` |
| 모든 수취인 | `for receiver in receivers` |
| 모든 파산자 | `for debtor in debtors` |
| 모든 규칙 | `for rule in fraud_rules` |

### 8.5 결과와 화면을 정의한다

| 결과 | 코드 표현 |
|---|---|
| 의심 여부 | `is_suspicious` |
| 위험 점수 | `risk_score` |
| 의심 유형 | `risk_type` |
| 근거 문장 | `reason` |
| 관련 거래 목록 | `evidence_transactions` |
| 관재인용 보고 | `report` |

마지막으로 사용자가 무엇을 볼지 정한다. Streamlit 대시보드라면 위험 점수 카드, 의심 유형 표, 거래 타임라인, 수취인별 송금 그래프, 근거 거래 테이블, 설명 문장이 필요하다.

## 10. 파산관재인 FDS 사고틀 예시

파산관재인 FDS를 만든다는 것은 법률적·금융적 의심 정황을 데이터 구조로 담고, 규칙과 알고리즘으로 판단하며, 그 판단 근거를 관재인이 이해할 수 있는 설명과 화면으로 제공하는 일이다.

## 11. 문법을 현실 문제와 연결하는 7단계 훈련 루틴

문법을 배울 때마다 다음 7단계를 적용하면 문법 지식이 프로젝트 부품으로 전환된다.

| 단계 | 훈련 내용 | 예시 |
|---|---|---|
| 1단계 | 문법의 정의를 한 문장으로 말한다 | 딕셔너리는 key-value 구조이다 |
| 2단계 | 현실 비유를 붙인다 | 회원번호로 회원 정보를 찾는 명부이다 |
| 3단계 | 작은 코드를 만든다 | `user = {"name": "홍길동", "points": 7500}` |
| 4단계 | 상태 변화를 추적한다 | `user["points"] += 1000` |
| 5단계 | 현실 문제에 연결한다 | 거래 위험 점수 누적 |
| 6단계 | 함수로 묶는다 | `add_risk_score(debtor, score)` |
| 7단계 | 프로젝트에 적용한다 | 파산자별 의심 점수 갱신 기능 |

```python
debtor = {
    "debtor_id": "D001",
    "name": "홍길동",
    "risk_score": 0,
    "suspicious_count": 0
}

def mark_suspicious(debtor, score):
    debtor["risk_score"] += score
    debtor["suspicious_count"] += 1
    return debtor
```

## 13. 좋은 코드의 기준

숙련된 프로그래머는 코드가 돌아가는지만 보지 않는다. 이름, 책임, 상태 변화, 자료구조, 확장성, 설명 가능성, 사용자 이해 가능성을 함께 본다.

| 질문 | 의미 |
|---|---|
| 이름이 정확한가 | 현실 개념을 잘 표현했는가 |
| 책임이 분리되어 있는가 | 한 함수가 너무 많은 일을 하지 않는가 |
| 상태 변화가 명확한가 | 값이 언제 어떻게 바뀌는지 보이는가 |
| 자료구조가 적절한가 | 문제를 풀기 좋은 형태로 데이터를 담았는가 |
| 확장 가능한가 | 규칙이 늘어나도 구조가 버티는가 |
| 설명 가능한가 | 왜 이런 결과가 나왔는지 말할 수 있는가 |
| 사용자가 이해할 수 있는가 | 결과가 업무 판단으로 이어지는가 |

FDS 프로젝트에서 가장 중요한 기준은 설명 가능성이다. 단순히 "위험도 87%"라고 말하는 시스템은 부족하다. 좋은 FDS는 위험 판단의 근거를 사람이 이해할 수 있는 문장과 증거 데이터로 함께 제시해야 한다.

**예시 설명**: 파산 신청 전 30일 동안 동일 수취인 김OO에게 7회 송금되었고, 총액이 6,300,000원이다. 각 거래는 100만 원 이하로 분산되어 있으나, 합산 금액과 반복성을 고려할 때 쪼개기 송금 가능성이 있다.

## 14. 데이터 분석가와 FDS 개발자를 위한 학습 방향

데이터 분석가 취업과 파산관재인 FDS 창업을 목표로 한다면 일반적인 방식으로 오래 문법만 공부하면 안 된다. 모든 문법은 실제 프로젝트의 기능으로 전환되어야 한다.

```
문법 학습
    ↓
작은 예제
    ↓
현실 문제 번역
    ↓
FDS 미니 기능 구현
    ↓
GitHub 기록
    ↓
Streamlit 시각화
    ↓
포트폴리오 설명 문서화
```

| 문법 | 파산관재인 FDS | 은행권 FDS | 커머스 데이터 분석 |
|---|---|---|---|
| 리스트 | 거래 목록 | 카드 사용 내역 | 주문 목록 |
| 딕셔너리 | 수취인별 송금액 | 고객별 위험 점수 | 상품별 매출 |
| 조건문 | 의심 기준 | 이상 거래 탐지 | 이탈 고객 조건 |
| 반복문 | 전체 거래 검사 | 전체 고객 스캔 | 전체 주문 분석 |
| 함수 | 위험 계산 | 스코어링 로직 | KPI 계산 |
| 클래스 | 파산자·거래 모델 | 고객·계좌 모델 | 유저·상품 모델 |
| DataFrame | 거래 분석 | 로그 분석 | 매출·전환 분석 |
| Streamlit | 관재인 대시보드 | 리스크 모니터링 | 비즈니스 리포트 |

학습의 기준 질문은 "이 문법을 어디에 쓸 수 있는가"가 되어야 한다. 파산관재인 FDS, 은행권 FDS, 쿠팡·무신사 데이터 분석 업무 중 하나에 연결되지 않는 문법 학습은 우선순위가 낮다.

## 15. 최종 정리 및 확인 질문

프로그래밍 문법을 배우는 목적은 문법 자체를 외우는 것이 아니다. 목적은 현실 문제를 컴퓨터가 처리할 수 있는 구조로 번역하는 능력을 얻는 것이다.

| 구분 | 핵심 정리 |
|---|---|
| 목적 | 현실 문제를 컴퓨터가 처리할 수 있는 구조로 번역하는 능력 획득 |
| 방향 | 문법 의미 이해에서 상태 변화, 자료구조, 함수, 클래스, 인터페이스로 확장 |
| 방법 | 문법 하나를 현실 문제와 프로젝트 기능으로 연결 |
| 산출물 | 코드, 설명 문장, 표, 그래프, 대시보드, 보고서 |
| 최종 목표 | 설명 가능한 FDS 및 데이터 분석 포트폴리오 구축 |

파산관재인 FDS를 만든다는 것은 법률적·금융적 의심 정황을 데이터 구조로 담고, 규칙과 알고리즘으로 판단하며, 그 판단 근거를 관재인이 이해할 수 있는 설명과 화면으로 제공하는 일이다.

다음 질문에 답할 수 있으면 단순히 파이썬 문법을 공부하는 단계가 아니라 현실 문제를 소프트웨어로 번역하는 설계자 단계로 들어선 것이다.

**확인 질문**

- 쪼개기 송금 의심이라는 현실 개념에서 무엇을 데이터로 담을 것인가
- 무엇을 조건으로 판단할 것인가
- 무엇을 반복해서 검사할 것인가
- 무엇을 함수로 묶을 것인가
- 무엇을 클래스로 만들 것인가
- 최종적으로 관재인에게 어떤 설명 문장으로 보여줄 것인가
- 이 결과를 어떤 화면과 그래프로 제시할 것인가
